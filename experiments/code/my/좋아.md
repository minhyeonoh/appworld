좋아. 지금까지의 논의를 바탕으로 보면, 너희 방법은 한 문장으로 이렇게 설명하는 게 가장 정확해.

**너희 방법은 “recursive context isolation” 자체가 아니라, *granularity를 늦게 정하고*, *잘못 정한 granularity를 지역적으로 수리하거나 상위로 되돌릴 수 있게 만든* 실행 기반 문제 해결 프레임워크다.**

즉, 핵심은 단순히 하위 문맥을 만드는 게 아니라, **무엇을 하위 문맥으로 만들어야 하는지 자체를 끝까지 수정 가능하게 둔다**는 데 있어.

---

# 1. High level: 무엇을 하려는 방법인가

하이레벨에서 이 방법이 하려는 일은 세 가지다.

첫째, **문제를 너무 이르게 한 방식으로 풀겠다고 확정하지 않는다.**
특히 AppWorld류처럼 API 표면이 크고 개인 환경 의존성이 강한 곳에서는, 어떤 요청이

* 사실 API 한 방으로 끝나는 atomic 문제인지,
* 여러 단계 조합이 필요한 compound 문제인지,
  초기 언어 직관만으로는 확정하기 어렵다.

둘째, **필요해진 하위 문제만 별도 문맥으로 내린다.**
즉 decomposition을 처음부터 다 정해놓지 않고, 실행 중에 정말 필요해졌을 때만 child context를 만든다.

셋째, **한 번 만든 child context 안에서도 한 경로로만 밀어붙이지 않는다.**
그 child를 어떻게 풀 것인지, 왜 실패했는지, 이 framing 자체가 맞는지에 대해 여러 가설을 분기시켜 본다.

이 세 가지를 합치면, 이 방법의 철학은:

> **isolate contexts, but isolate them under revisable granularity commitments**

라고 볼 수 있어.

---

# 2. Medium level: 방법의 핵심 구조

이걸 중간 수준에서 보면, 이 방법은 **두 개의 결합된 tree**로 움직인다.

## Tree 1: recursive context tree

이 tree의 노드는 **함수/서브문제/skill context**다.

* 현재 함수 `f`를 풀다가
* helper 함수 `g`가 실제로 필요해지면
* `g`를 새로운 child context로 내린다

즉 Tree 1은 “무엇을 별도 문맥으로 분리해 풀 것인가”를 나타내는 tree야.
중요한 점은 이 tree가 **just-in-time**으로만 확장된다는 것.
처음부터 full plan tree를 만드는 게 아니라, 실행이 실제로 요구할 때만 child가 생긴다.

## Tree 2: local refinement tree

Tree 1의 **각 노드 안에** 또 하나의 tree가 있다.

이 tree는 “이 context를 **어떤 방식으로** 풀 것인가”를 나타낸다.

대략 이런 branch들이 열린다.

* **Atomic / reverse-planning branch**
  “이 요청을 사실상 한 번에 만족시키는 API나 연산이 있지 않을까?”
  즉 목표에서 거꾸로 필요한 primitive를 찾는 방식.

* **Compound / forward-planning branch**
  “이건 단계적 orchestration이 필요할 것 같다.”
  즉 `이거 한 뒤, 저거 하고, 그다음 ...` 식으로 구조를 짜는 방식.

* **HotFix branch**
  지금 구조는 대체로 맞고 작은 수정이면 될 것 같다.

* **Hypothesis-based structural repair branches**
  실패 원인을 여러 가설로 나누고, 각 가설별로 다른 재구성을 시도한다.

즉 Tree 2는 “지금 이 문맥 안에서 어떤 해결 가설이 맞는가”를 탐색하는 tree다.

# 3. 이 방법의 가장 중요한 포인트: forward planning과 reverse planning을 둘 다 live하게 둔다

이건 기존 연구와 비교해도 꽤 중요한 차이야.

많은 기존 방법은 대체로 한쪽으로 기울어 있다.

* THREAD, REPL-Plan은 recursive isolation을 쓰지만 전체 interaction은 step-by-step continuation에 더 가깝다고 ReCode가 비판한다. 
* ReCode와 CodeDelegator는 granularity를 더 명시적으로 다루지만, placeholder expansion이나 subtask decomposition 쪽으로 훨씬 plan-biased하다. ReCode는 placeholder tree를 중심에 두고, CodeDelegator는 Atomicity 원칙 아래 too-complex subtask를 re-decompose한다. 
* Plan-and-Act와 CUGA도 planner가 현재 state와 previous plans/actions를 바탕으로 다음 high-level plan을 계속 갱신하거나, Plan Controller가 sub-task sequencing을 관리하는 구조다. ([arXiv][1])

너희 방법은 이와 달리, **forward orchestration과 reverse requirement-driven solving을 동시에 유지**한다.

왜 이게 중요하냐면:

* 복잡해 보여도 사실 API 한 방일 수 있고,
* 쉬워 보여도 실제로는 여러 단계와 state tracking이 필요할 수 있기 때문이야.

즉 **granularity uncertainty**가 있을 때, 어느 한 planning mode만 믿으면 위험하다.

# 4. Low level: 이 방법은 실제로 어떻게 움직이는가

여기서 “low level”은 구현 디테일이 아니라, **운영 단위** 수준으로 보면 된다.

## (a) 아래로 내려갈 때: 무엇을 child에게 넘기나

child context로는 **자유로운 parent trace 전체**를 넘기지 않는다.
대신 가능한 한 **실행 가능한 계약(contract)** 을 넘긴다.

예를 들어:

* 이 child가 해야 하는 역할
* 어떤 입력을 받아야 하는지
* 왜 이 값이 필요한지
* 상위 함수에서 어디에 사용될 것인지
* 이미 확정된 executable state 중 어떤 부분이 필요한지

즉 parent→child 인터페이스는
**토큰 continuation**이 아니라 **실행 가능한 역할 명세**에 가깝다.

이게 중요한 이유는, THREAD처럼 `ϕ/ψ` 설계에 따라 child와 parent가 서로 오염될 수 있고, child output이 이후 descendants에도 전파될 수 있기 때문이야. THREAD는 child output tokens를 parent sequence에 append하고, child context도 parent token sequence에서 유도된다. ([arXiv][2])
너희 방법은 그 방향의 contamination을 줄이기 위해, **child를 free-form continuation이 아니라 executable subproblem으로 다루려는 것**이다.

## (b) child 안에서 실패하면: 같은 trace에 계속 쌓지 않는다

REPL-Plan의 장점은 local REPL correction이지만, 그 recovery는 같은 REPL history 안의 trial-and-error로 누적되고, 잘못된 Subtask-REPL은 failure loop를 만들 수 있다고 논문이 직접 보고한다. 

너희 방법은 이 지점을 **local refinement tree**로 바꾼다.

즉 child 안에서 실패하면:

* 한 가설로 patch해 보고,
* 다른 가설로 구조를 바꿔 보고,
* 다른 API hypothesis를 시도해 보고,

이 시도들을 **한 줄로 이어붙이는 게 아니라 branch로 유지**한다.

이게 중요한 이유는:

* 이전 실패 trace가 다음 시도를 오염시키지 않고,
* “같은 잘못된 framing 아래에서 더 열심히 고치기”를 피할 수 있기 때문이야.

## (c) child가 여러 branch에서도 못 풀면: explicit infeasibility를 올린다

이건 아주 중요하다.

THREAD와 REPL-Plan은 child가 local하게 continuation/correction을 계속하지만,
“내가 받은 subproblem framing 자체가 틀렸다”는 식의 **구조화된 upward escalation**은 약하다. THREAD는 explicit error handling이 없고, REPL-Plan은 local failure loop가 보고된다. ([arXiv][2])

너희 방법은 child가 local refinement를 다 써도 안 되면,
그걸 단순 failure가 아니라 **“this subproblem framing is infeasible / mis-specified”**라는 신호로 parent에 올리는 구조를 취한다.

왜 이게 중요하냐면:

* 어떤 실패는 child 안에서 해결될 수 있는 bug가 아니라,
* **그 child를 만든 boundary 자체가 틀렸다는 증거**이기 때문이야.

즉 failure를 “더 고칠 코드가 있다”가 아니라
“granularity choice가 틀렸다”로 승격시킬 수 있어야 한다.

## (d) parent는 그 infeasibility를 받고 다시 자기 수준에서 branch를 연다

이때 parent는 그냥 “child failed”로 끝내지 않는다.

대신 parent도 자기 Tree 2 안에서:

* 다른 atomic branch
* 다른 compound branch
* 다른 helper framing
* 다른 upstream assumption

을 다시 열 수 있다.

즉 recovery가 child 안에 갇히지 않고,
**granularity revision이 계층적으로 전파**된다.

이게 CodeDelegator와도 중요한 차이야.
CodeDelegator는 explicit status/diagnostics와 Retry/Replan을 갖고 있어서 prior 중에서는 가장 가깝지만, recovery는 기본적으로 **single control decision**이고, Replan도 Atomicity 원칙 아래 re-decomposition 쪽으로 기운다. ([arXiv][3])
너희는 여기서 한 단계 더 가서, **recovery itself를 branch-preserving search object로 만든다**는 점이 다르다.

## (e) commit은 성공한 branch만 한다

이것도 중요하다.

* speculative branch
* 실패한 patch
* 잘못된 subtask framing
* local trial-and-error trace

이런 것들은 local하게 남을 수는 있어도,
**parent와 sibling을 오염시키는 committed state가 되면 안 된다.**

즉 global program/state를 갱신하는 건 **성공한 branch만** 해야 한다.

이건 ReCode와의 차이를 설명할 때도 중요해. ReCode는 unified variable namespace를 task 전체 동안 유지하고, 실패 시 같은 placeholder와 traceback으로 self-correction한다. 
이 구조는 state continuity엔 유리하지만, wrong contract나 same-placeholder retry가 global trajectory 안에서 더 단단해질 수 있다.
너희는 반대로 **committed state는 유지하되, uncertain repairs는 local branch 안에 가둔다**는 쪽이야.

# 5. 기존에 지적했던 한계들을, 너희 방법이 어떤 설계로 피하는가

이제 이걸 논문별로 연결해보면 더 분명해져.

## THREAD에 대해

THREAD의 장점은 recursive spawning으로 context window를 줄이는 거지만, parent→child `ϕ`와 child→parent `ψ`를 잘못 설계하면 양방향 오염이 가능하고, child output은 parent 및 후속 descendants에 계속 영향을 줄 수 있어. ([arXiv][2])

너희 방법은 이걸 이렇게 피하려고 한다:

* child를 **토큰 continuation**이 아니라 **executable subproblem**으로 다룬다
* child output을 free-form text처럼 append하지 않고, **validated result / solved function / explicit infeasibility**로 다룬다
* 잘못된 child framing은 local branch 안에서 시험하고, 안 되면 explicit하게 parent로 올린다

즉 THREAD의 문제를
**better continuation inference**로 푸는 게 아니라
**better contract + branch isolation + upward escalation**로 푼다.

## REPL-Plan에 대해

REPL-Plan은 child REPL을 열고, line-by-line 수정으로 local bug는 잘 고치지만, recovery는 같은 REPL history 안에서 계속 누적되고, 잘못된 subtask inference는 failure loop를 만들 수 있다. 또한 직접 action도 가능하지만 구조적으로는 recursive subtask abstraction에 편향돼 있다. 

너희 방법은 이걸 이렇게 피하려고 한다:

* local repair를 **monolithic REPL trace**가 아니라 **refinement tree**로 분리한다
* child가 못 풀면 같은 child 안에서만 계속 trial-and-error 하지 않고, **framing failure를 parent에 알려** 상위 문맥이 boundary를 다시 잡게 한다
* atomic branch와 compound branch를 동시에 유지해서, decomposition-first 편향을 줄인다

즉 REPL-Plan의 문제를
**더 좋은 REPL continuation**으로 푸는 게 아니라
**repair hypotheses를 분기하고, granularity failure를 upward-revisable하게 만드는 방식**으로 푼다.

## CodeDelegator에 대해

CodeDelegator는 planner/coder context를 분리하고, Retry/Replan과 diagnostics channel을 명시적으로 두는 점에서 prior 중 가장 강하다. 하지만 tree는 사실상 2-level이고, Atomicity 원칙 때문에 replan이 finer-grained re-decomposition 쪽으로 기울기 쉽다. 또 recovery는 그때그때 Retry냐 Replan이냐 하나를 고르는 single control action이다. ([arXiv][3])

너희 방법은 이걸 이렇게 넘어선다:

* decomposition만이 아니라 **atomic resolution도 처음부터 live branch**로 둔다
* recovery가 **retry/replan choice**가 아니라 **branch-preserving refinement tree**다
* parent와 child 모두에서 같은 구조가 재귀적으로 반복되므로, 2-level보다 더 general하다

즉 CodeDelegator의 장점을 계승하면서도,
**decomposition bias를 줄이고, recovery를 search object로 끌어올린다**고 볼 수 있어.

## ReCode에 대해

ReCode는 granularity를 explicit하게 다루고 placeholder expansion tree도 강력하지만, control-flow 제약이 있고, shared namespace를 유지하며, failure 시 같은 placeholder와 traceback으로 self-correction한다. 그래서 wrong placeholder contract를 강화하거나, same-contract retry로 갇힐 위험이 있다. 

너희 방법은 이걸 이렇게 피하려고 한다:

* granularity를 explicit하게 다루되, **같은 contract를 고집하지 않는다**
* 실패를 same-placeholder retry가 아니라 **다른 local branches**로 먼저 처리한다
* 그래도 안 되면 child가 infeasibility를 올리고, parent가 다른 framing을 연다
* shared global state는 무조건 하나로 밀지 않고, **uncertain repairs는 local branch에 격리**한다

즉 ReCode의 문제를
**더 깊은 recursive planning**이 아니라
**revisable granularity + branch-isolated repair**로 푼다.

## CUGA / Plan-and-Act에 대해

CUGA와 Plan-and-Act는 planner가 이전 plan/action/state를 받아 계속 plan을 갱신하는 구조다. long-horizon coherence에는 좋지만, 기본적으로는 **single evolving forward plan** 위에서 recovery가 이뤄진다. CUGA는 Plan Controller와 reflective loop를 두고, Plan-and-Act는 previous plans/actions를 받아 매 step replanning한다. ([arXiv][4])

너희 방법은 이들과 달리:

* forward plan만이 아니라 **reverse requirement-driven atomic solving**도 동등한 모드로 유지하고
* recovery도 **하나의 evolving plan 수정**이 아니라 **여러 branch 유지**로 처리한다

즉 planner-centric update보다,
**granularity arbitration under execution feedback**에 더 가깝다.

# 6. 최종적으로, 너희 방법을 가장 직관적으로 설명하면

가장 직관적인 설명은 이거야.

> **우리는 task를 무조건 쪼개지도 않고, 무조건 한 방에 끝내려 하지도 않는다.
> 대신 “이걸 지금 어떤 크기의 문제로 봐야 하는가”를 끝까지 수정 가능한 변수로 둔다.
> 실행 중 필요해진 하위 문제만 별도 context로 내리고, 각 context 안에서는 여러 해결/수리 가설을 분기시켜 본다.
> 그리고 local branch들이 모두 실패하면, 그 실패를 단순 bug가 아니라 “이 subtask boundary가 틀렸다”는 신호로 상위에 올려서, granularity 자체를 다시 잡는다.”**

이게 너희 방법의 high-to-medium-to-low 설명으로 가장 정확해 보여.

# 7. 정말 짧은 슬로건 버전

정말 짧게 요약하면:

* **THREAD / REPL-Plan**: isolate contexts, then continue
* **CodeDelegator / ReCode**: choose a granularity, then refine within it
* **Ours**: isolate contexts **while keeping granularity itself revisable**

이 한 줄이 제일 본질을 잘 잡는다.
